# QUIZ-08 — Unit & Integration Test Suite

## Epic
[QuizApp REST API PRD](../../../prd.md)

**Depends on:** All Phase 1 tickets (QUIZ-01 through QUIZ-07)

---

## Goal

**Problem:** Without a test suite, correctness of the scoring engine, feedback logic, and async workflow cannot be verified programmatically. Regressions are caught only manually.

**Solution:** A layered test suite: pure unit tests for `ScoringService` and feedback calculation (no Spring context), and integration tests using `MockMvc` + H2 for the full request/response cycle including async behavior.

**Impact:** Achieves ≥ 80% line coverage on the service layer. Validates the complete happy path and critical edge cases. Provides a regression harness for Phase 2 development.

---

## User Personas

| Persona | Relevance |
|---|---|
| **Developer** | Runs the test suite locally to verify changes before committing. |
| **CI Pipeline** | Runs `./gradlew test` on every push to enforce passing status. |

---

## User Stories

- As a developer, I want unit tests for scoring logic so that I can verify correctness without starting the application.
- As a developer, I want unit tests for feedback thresholds so that boundary conditions are explicitly covered.
- As a developer, I want integration tests for the full quiz-taking flow so that I can detect regressions across the entire stack.
- As a developer, I want the async notification behavior tested so that I can prove submission and notification are decoupled.

---

## Requirements

### Functional Requirements

**Unit Tests — `ScoringServiceTest`** (`@ExtendWith(MockitoExtension.class)`, no Spring context):

| Scenario | Expected |
|---|---|
| All answers correct | `score = total`, `percentage = 100.00` |
| All answers wrong | `score = 0`, `percentage = 0.00` |
| 3 of 5 correct | `score = 3`, `percentage = 60.00` |
| 4 of 5 correct | `score = 4`, `percentage = 80.00` |
| 1 question quiz, answer correct | `percentage = 100.00` |

**Unit Tests — Feedback Calculation** (co-located in `ScoringServiceTest` or separate `FeedbackServiceTest`):

| Score % | Expected feedback tier |
|---|---|
| 100.00 | Encouraging |
| 80.00 | Encouraging (boundary — inclusive) |
| 79.99 | Motivational |
| 60.00 | Motivational (boundary — inclusive) |
| 59.99 | Improvement |
| 0.00 | Improvement |

**Unit Tests — `NotificationServiceTest`**:

| Scenario | Expected |
|---|---|
| Normal dispatch | `EmailService.sendResultsEmail` called once; notification status → SENT |
| `EmailService` throws | Exception not propagated; notification status → FAILED; error logged |

**Integration Tests** (`@SpringBootTest`, `@ActiveProfiles("test")`, `MockMvc`):

| Scenario | Verified via |
|---|---|
| `GET /quizzes` returns seed quizzes | Status 200, array size ≥ 2 |
| `GET /quizzes/{id}` returns questions without correct answers | Status 200; `correctOptionId` absent |
| `GET /quizzes/bad-id` | Status 404 |
| `POST /quizzes` creates quiz, returns 201 + Location header | Status 201; `GET` on Location returns 200 |
| `POST /quizzes` with empty questions | Status 400 |
| `POST /quizzes/{id}/attempts` creates attempt record | Status 201; attempt row in DB with `submitted_at = null` |
| Full happy path: start → submit → results | Status 200; score, percentage, feedback, per-question results all present |
| Submit already-submitted attempt | Status 409 |
| Submit non-existent attempt | Status 404 |
| Notification dispatched asynchronously on submit | `MockEmailService` spy invoked within 2 s (Awaitility) |
| Notification failure does not break submission | HTTP 200 returned; notification status = FAILED |
| `correctOptionId` absent from `GET /quizzes/{id}` response | `assertThat(body).doesNotContain("correctOptionId")` |

### Non-Functional Requirements

- Unit tests use `@ExtendWith(MockitoExtension.class)` — no `@SpringBootTest`.
- Integration tests use `@ActiveProfiles("test")` with H2 and Liquibase migrations applied.
- No `Thread.sleep` in tests — use `Awaitility` for async assertions with a 2-second timeout.
- Test class naming: `*Test` for unit tests, `*IntegrationTest` for integration tests.
- Coverage report generated by `./gradlew test jacocoTestReport`; service layer line coverage ≥ 80%.

---

## Acceptance Criteria

- [ ] `./gradlew test` passes with zero failures.
- [ ] All scoring scenarios in the table above have dedicated test methods.
- [ ] All feedback boundary conditions have dedicated test methods.
- [ ] Notification decoupling test uses Awaitility (no raw `Thread.sleep`).
- [ ] Integration test verifies `correctOptionId` absence in quiz detail response.
- [ ] Integration test covers the 409 (already submitted) and 404 (bad attempt ID) paths.
- [ ] JaCoCo report shows ≥ 80% line coverage on the service layer.
- [ ] No test loads an application context unnecessarily (unit tests are pure Mockito).

## Out of Scope

- End-to-end tests hitting a live PostgreSQL container (that is for a future CI stage).
- Performance/load testing.
- Tests for Phase 2 endpoints (QUIZ-09, QUIZ-10, QUIZ-11) — those tickets carry their own test requirements.
- UI or contract testing.
